Auto-completar el F.572 con una extensión de Chrome

Vos te logueás a SiRADIG con tu Clave Fiscal como siempre. La extensión, parada sobre esa página ya autenticada, lee tus facturas desde arcaInvoices y escribe los campos del formulario por vos, una deducción a la vez, dejándote revisar y confirmar. Nunca toca tu Clave Fiscal.

Cómo se siente al usarla

1 · LoginEntrás a SiRADIG con Clave Fiscal (vos, a mano; 2FA incluido).
→
2 · PanelLa extensión muestra las facturas pendientes de arcaInvoices.
→
3 · "Cargar siguiente"Rellena CUIT, comprobante, fecha, importe y categoría.
→
4 · Revisás y confirmásVos apretás "Guardar" de AFIP. Se marca como cargada.

El humano siempre da el último clic de guardar: mantiene el control, evita cargar datos errados y nos deja del lado seguro de los Términos de AFIP.

Arquitectura (Manifest V3)

Content script

Inyectado en las páginas 572web. Conoce el mapa del DOM y escribe los campos disparando los eventos correctos.

Service worker

Trae las facturas desde la API de arcaInvoices, mantiene la cola y mensajea con el panel.

Side panel / popup

Lista de pendientes, botones "cargar / saltar / marcar cargada", progreso del mes.

Permisos mínimos: host_permissions acotados a los dominios del 572 de AFIP y al dominio de tu app. Nada de <all_urls>.

El puente de datos · cómo llegan las facturas a la extensión

La extensión necesita leer /api/export/summary de arcaInvoices. Hay tres formas; la cookie de sesión es SameSite=Lax, así que un fetch cruzado directo no la manda. Opciones:

EnfoqueCómoVeredicto
Token de conexiónrecomendadoLa app genera un token de API de alcance limitado; lo pegás una vez en la extensión y queda guardado.Simple, seguro, sin tocar cookies
Cookie cross-siteCambiar la cookie a SameSite=None; Secure + CORS para el origen de la extensión.Funciona, pero afloja la cookie
Leer desde la pestañaEl usuario abre arcaInvoices en otra pestaña; la extensión lee los datos vía mensajería.Más fricción de uso

La parte difícil · el auto-fill sobre un formulario legacy

El 572web es un formulario viejo con postbacks (ASP/JSF): el alta de una deducción es un asistente de varios pasos y muchos campos recargan parte de la página al cambiar. Por eso no alcanza con setear input.value: hay que disparar los eventos input/change/blur, esperar a que el postback asiente, y recién seguir. Es la parte frágil y la que más mantenimiento pide.

// Boceto del content script (simplificado)
async function fillDeduccion(inv) {
  setField('#cuitEmisor', inv.cuit);          // set value + dispatch input/change
  await settle();                              // esperar postback/AJAX
  setField('#nroComprobante', inv.invoice_number);
  setField('#fecha', inv.date);
  setField('#importe', inv.amount);
  selectDeduccion(mapCategoria(inv.category)); // arcaInvoices → tipo SiRADIG
  await settle();
  highlightConfirm();   // NO hace submit: resalta el botón "Guardar" de AFIP
}
  • Seleccionar campos por atributos estables y centralizar el "mapa del DOM" en un solo módulo versionado.
  • Tolerar campos ausentes y nunca hacer el submit final automático → menos riesgo de datos malos.
  • Si AFIP cambia el formulario, los selectores se rompen y hay que actualizar la extensión.

Seguridad

  • Cero Clave Fiscal: la extensión nunca ve tu contraseña; actuás sobre una página que ya autenticaste vos.
  • Permisos acotados: solo los dominios del 572 + el de tu app.
  • Datos efímeros: la cola de facturas vive en memoria/sesión; el token de conexión es de alcance limitado y revocable.
  • Mantener al humano en el guardado final también ayuda a la postura frente a los Términos de AFIP.

Distribución y mantenimiento

Chrome Web Store multi-usuario

Auto-update y fácil para los usuarios que invites. Esperá una revisión por los permisos sobre dominios de AFIP.

Cargar desempaquetada

Para vos/desarrollo: sin revisión, pero cada uno la instala a mano.

CRX self-hosted

Intermedio; control total de la actualización.

Mantenimiento: está acoplada al DOM de AFIP. Mitigación: mapa del DOM centralizado y versionado, y el co-piloto "smart paste" como red de seguridad si algo se rompe a mitad de mes.

Plan por fases y esfuerzo

FaseQué incluyeEsfuerzo
A · SpikeCapturar el DOM real del asistente (sesión logueada) y rellenar una deducción de punta a punta, resolviendo los postbacks. Valida factibilidad.~1–2 días
B · MVPPuente de datos (token + /export/summary), cola en el panel, mapeo categoría→tipo de deducción, auto-fill de todas las facturas.~1 semana
C · PulidoProgreso, "saltar / marcar cargada" que escribe el sync_status de vuelta en arcaInvoices, empaquetado y envío a la Web Store.~2–3 semanas (incl. revisión)

Estimaciones aproximadas; la incógnita más grande es cuánto "pelea" el formulario con los postbacks, y eso lo despeja la Fase A.

Riesgos y mitigaciones

  • Cambios de UI de AFIP → mapa del DOM centralizado + versionado; fallback a smart-paste.
  • Términos de AFIP → login humano, sin credenciales guardadas, sin submit automático.
  • Cookie/CORS → usar token de conexión en vez de cookie cruzada.
  • Revisión de la Web Store → justificar los host_permissions; mientras tanto, "cargar desempaquetada".

Recomendación + qué necesito definir

Arrancaría por la Fase A (spike): es barata y nos dice de inmediato si el formulario se deja automatizar bien. Si sale, seguimos al MVP.

Decisiones para avanzar:

  1. Puente de datos: ¿token de conexión (recomendado), cookie cross-site, o leer desde la pestaña?
  2. Alcance del auto-fill: ¿solo rellenar y que vos avances los pasos, o que también avance el asistente?
  3. Distribución: ¿solo para vos (desempaquetada) o multi-usuario (Web Store)?
  4. ¿Arrancamos por el spike (Fase A)? Para eso necesito acceso a una sesión real del 572 para mapear el DOM.
